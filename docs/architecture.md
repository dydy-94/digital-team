# 多智能体沙箱群聊系统 - 系统架构文档

## 1. 项目概述

### 1.1 项目目标
构建一个支持多智能体（Agent）在群聊环境中进行自主协作的沙箱系统。用户可以在聊天室内与多个 AI Agent 进行对话，Agent 之间可以相互 @ 进行协作。

### 1.2 核心能力
- **群聊管理**：创建、加入、删除聊天室
- **消息广播**：支持 @ 提及特定 Agent
- **发言控制**：基于令牌锁的发言冲突解决机制
- **历史消息**：持久化存储聊天记录，支持历史消息加载
- **Agent 协作**：Agent 可以通过 @ 互相唤醒进行协作

## 3. 数据流

### 3.1 消息发送流程

```
1. 用户在聊天室发送消息 (包含 @ 提及)
         │
         ▼
2. 消息通过 WebSocket 发送到 Coordinator
         │
         ▼
3. Coordinator 获取发言令牌锁
         │
         ▼
4. Coordinator 广播消息到所有聊天室成员
         │
         ├──► x-client (非被 @)
         │         │
         │         ▼
         │    旁听模式：存储到 Memory Window
         │
         └──► x-client (被 @)
                   │
                   ▼
              唤醒 AgentCore
                   │
                   ▼
              获取响应后发送回复到 Coordinator
                   │
                   ▼
5. Coordinator 释放发言令牌锁 (延迟释放)
```

### 3.2 消息数据结构

#### A2A Message (Agent to Agent)
```json
{
  "msgId": "uuid-string",
  "channelId": "room_xxx",
  "sender": "agent_1",
  "target": "ALL",
  "mentionUsers": ["agent_2"],
  "intent": "INFORM",
  "contentText": "消息内容",
  "timestamp": 1234567890
}
```

#### Client Message (发送到 Coordinator)
```json
{
  "action": "speak|join|leave|register",
  "data": { ... }
}
```

#### Server Message (从 Coordinator 接收)
```json
{
  "type": "message|history|error",
  "data": { ... }
}
```

## 4. 核心机制

### 4.1 发言控制 (Speaker Lock)

为防止多个 Agent 同时发言造成混乱，系统实现了基于令牌锁的发言控制机制：

- **锁获取**：消息发送前尝试获取锁，超时时间 2 秒
- **锁释放**：
  - 被 @ 提及的消息：立即释放锁
  - 普通消息：延迟 2 秒后释放锁
- **冲突处理**：返回 `reject` 消息给发送者

### 4.2 上下文管理 (Memory Window)

每个 x-client 维护每个聊天室的消息历史：

- **窗口大小**：默认 50 条消息
- **字符限制**：默认 2000 字符
- **上下文构建**：唤醒 AgentCore 时，将历史消息构建为上下文提示

### 4.3 自动重连

x-client 实现了自动重连机制：

- **心跳间隔**：默认 30 秒
- **重连间隔**：默认 5 秒
- **重连策略**：无限重试，直到连接成功

### 4.4 持久化策略

- **聊天室**：创建时持久化
- **成员**：加入/离开时更新
- **消息**：发送时持久化
- **历史加载**：成员加入聊天室时加载最近 50 条消息

## 5. 存储设计

### 5.1 数据库 Schema

#### rooms 表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(255) | 聊天室 ID (主键) |
| name | VARCHAR(255) | 聊天室名称 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

#### members 表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 (自增) |
| room_id | VARCHAR(255) | 聊天室 ID |
| agent_id | VARCHAR(255) | Agent ID |
| joined_at | DATETIME | 加入时间 |
| left_at | DATETIME | 离开时间 (NULL 表示仍在聊天室) |

#### messages 表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 (自增) |
| room_id | VARCHAR(255) | 聊天室 ID |
| msg_id | VARCHAR(255) | 消息 ID |
| sender | VARCHAR(255) | 发送者 |
| target | VARCHAR(255) | 目标 (ALL 表示所有人) |
| mention_users | TEXT | @ 提及的用户 (逗号分隔) |
| intent | VARCHAR(50) | 意图 |
| content_text | TEXT | 消息内容 |
| timestamp | BIGINT | 时间戳 |
| created_at | DATETIME | 创建时间 |

## 9. 安全性考虑

### 9.1 跨域
- WebSocket 允许所有来源 (`CheckOrigin: true`)
- 生产环境应限制来源

### 9.2 消息去重
- x-client 维护最近 100 条消息 ID
- 超过 100 条时清理一半

### 9.3 进程隔离
- 使用独立进程组 (`Setpgid`)
- 防止父进程退出影响子进程
